00

What is this thing?

You've seen AI chatbots confidently make things up. Provenas is built so that cannot happen.

Here's the trick. There are two parts, and they have very different jobs:

  • The talker (an AI language model) only gets to suggest things — "I think you're asking whether Alice can deploy to production."
  • The fact-keeper (a small, exact database of facts and rules) does the actual answering. It's like a calculator: it can't improvise, it can only compute. It is the only source of truth.

Think of it like a courtroom. The AI is a lawyer — persuasive, sometimes wrong. The fact-keeper is the court record. No matter how eloquent the lawyer is, nothing counts unless it's in the record, and every verdict cites the record line by line.

That design gives you two guarantees:

  • Every answer shows its work. Not just "yes" — but "yes, because of these exact facts and these exact rules," traced step by step like a math proof.
  • Nothing gets in without passing a test. If the AI suggests a new rule or writes a bit of code, Provenas tests the suggestion first. Fail the test → rejected, never saved. The AI can be wrong all day long and the record stays clean.

And if you ask something it simply doesn't know? It says "not derivable" — an honest "I can't conclude that" — instead of guessing. That one behavior is the whole point.

What would I use it for? Anything where "the AI probably got it right" isn't good enough: who has access to what (and why), who qualifies for a benefit or discount (and why), which parts are compatible, troubleshooting steps — any set of rules you currently keep in a PDF and three people's heads.
01

Install it, teach it facts

You need Python 3.10 or newer. There's nothing else to install — no accounts, no cloud.

in your terminalgit clone https://github.com/calvincs/Provenas && cd Provenas
pip install -e .
provenas mykb.db        # opens (or creates) your knowledge base — a single file

Everything Provenas knows is a fact, and a fact is just a three-word sentence: who — relationship — what. Like flash cards:

  • tom parent bob  means  "Tom is a parent of Bob."
  • alice has_role engineer  means  "Alice has the role engineer."
  • rosa age 71  means  "Rosa's age is 71."

Commands all start with a colon (:). Let's teach it two facts and ask about one. :why means "is this true — and show me why":

provenas — try thisprovenas> :assert tom parent bob
  added (tom parent bob)
provenas> :assert bob parent ann
  added (bob parent ann)
provenas> :facts
  (bob parent ann)
  (tom parent bob)
  [2 fact(s)]
provenas> :why tom parent bob
  yes
    (tom parent bob)   (given fact)

"Given fact" means: true because you told me so. In the next chapter, Provenas starts figuring out things you didn't tell it.

No AI needed yet Notice we haven't connected any AI model. Every :command works completely offline. The English-language layer (chapter 05) is an optional add-on — never a requirement, and never the authority.

Don't feel like typing facts? Load a ready-made set: :load family (a family tree), :load rbac (office access permissions), :load diagnostics (computer troubleshooting), :load eligibility (benefit rules), or :load config (compatible parts).

02

Rules: if-this-then-that

Facts are what you state. Rules are how Provenas figures out more. A rule is an if-then sentence:

"IF someone has a role, AND that role grants a permission — THEN that person has the permission."

In rules, words starting with ? are blanks to fill in — like the X in algebra. ?u can stand for any person, ?p for any permission. Load the office access pack and look at its three rules:

provenas — try thisprovenas> :load rbac
  loaded 'rbac'
provenas> :rules
  role-inherit: ?u has_role ?r2  ⇐  (?u has_role ?r), (?r inherits ?r2)
  perm-from-role: ?u has_perm ?p  ⇐  (?u has_role ?r), (?r grants ?p)
  can-do: ?u can ?a  ⇐  (?u has_perm ?p), (?a requires ?p)
  [3 active / 3 rule(s)]  (:disable/:enable <name>)

Read the ⇐ arrow as "whenever the right side holds." The middle rule says: whenever ?u has role ?r, and role ?r grants permission ?p — then ?u has permission ?p.

Provenas applies every rule to every fact, over and over, until there's nothing new left to conclude — then it saves all those conclusions, so asking questions later is instant. (Like solving a sudoku once and keeping the finished grid, instead of re-solving it every time someone asks what's in row 3.)

Now the payoff. Ask about something you never directly stated, and watch it show its work:

provenas — try thisprovenas> :why alice can prod_deploy
  yes
    (alice can prod_deploy)   ⇐ rule[can-do]
      (alice has_perm deploy_perm)   ⇐ rule[perm-from-role]
        (alice has_role engineer)   (given fact)
        (engineer grants deploy_perm)   (given fact)
      (prod_deploy requires deploy_perm)   (given fact)

Read it from the bottom up, like a receipt:

  1. Alice is an engineer (you said so), and engineers get the deploy permission (you said so) → so Alice has the deploy permission (rule).
  2. Deploying to production requires that permission (you said so) → so Alice can deploy (rule).

That receipt is the difference between "trust me" and "check me." Auditors, regulators, and anyone who's ever been burned by a confident wrong answer will know which one they want.

It's fast This isn't a toy speed-wise: 50,000 starting facts grow to a 287,000-fact knowledge base in about 1.3 seconds, and questions against the saved result come back in under a millisecond. There's a test in the repo that fails if it ever gets slower.
03

"Unless" and numbers

Real-world rules are rarely just "if A and B." They say "unless", and they compare numbers. Both work here. A rule's IF-part can use three kinds of condition:

conditionplain English
["?u", "has_role", "?r"] A lookup. "?u has some role ?r" — matched against the known facts.
["?u", "~suspended", "?x"] An "unless." The ~ means no such fact exists — "?u is not suspended, in any way, by anyone." If even one matching fact exists, the rule doesn't apply.
["?age", ">=", "65"] A number check. Also < <= > == !=. When both sides are numbers they compare as numbers (so 9 < 18, even though alphabetically "9" comes after "18").

Two real rules, exactly as a policy document would phrase them:

two rules, written out# "You're ALLOWED to do something if you CAN do it — unless you're suspended."
allowed:  ?u allowed ?a  ⇐  (?u can ?a), (?u ~suspended ?when)

# "Age 65 or older? You're eligible for the senior discount."
senior:   ?p eligible_for senior_discount  ⇐  (?p age ?a), (?a >= 65)

Here's the "unless" doing its job in a real run. Dev Dan is a contractor whose access was suspended on June 2nd. He still has the permission on paper — but he's not allowed:

examples/access_audit.py — real output[audit] Is the contractor CAPABLE of pushing to main?  (raw permission)
        -> yes
[audit] ...but is he ALLOWED?  (suspension wins — no special case needed)
        -> no
        (dev_dan allowed main_branch) is not derivable from the facts + rules.

Notice nobody wrote special "suspension-handling code." It's just a rule with an "unless," and the engine treats it like any other rule.

One guardrail you get for free Circular "unless" logic — like "A counts unless B, and B counts unless A" — has no sensible answer, so Provenas refuses rules that create that kind of loop rather than guessing.
04

Safety rails

This chapter is what separates "neat demo" from "thing you can actually rely on." Four rails:

1. Pin an answer, and no rule change can quietly flip it

Suppose today the system correctly says Alice can deploy. Pin that answer with :case — like photographing the scoreboard:

provenas — try thisprovenas> :why alice can prod_deploy
  yes …the receipt…
provenas> :case alice-prod
  pinned case 'alice-prod': {'action': 'check', 'triple': ['alice', 'can', 'prod_deploy']} -> True
  (rule changes must preserve it)

From now on, every proposed rule change gets re-checked against your pinned answers first. A new rule that would flip one — even accidentally — is rejected on the spot. Software engineers call this idea "regression testing"; here it guards your decisions.

2. A spell-checker for facts

Typos are how databases rot. Turn on strict mode and Provenas only accepts facts whose relationship word has been officially declared:

provenas — try thisprovenas> :strict on
  strict mode ON — asserts with undeclared relations are rejected
provenas> :assert bob hsa_role intern
  error: relation 'hsa_role' is not declared in the schema (strict mode)

(hsa_role was a typo for has_role — caught instead of silently stored.) The ready-made packs declare their words automatically; declare your own with :declare has_role "person holds role".

3. Rules can be switched off without being deleted

provenas — try thisprovenas> :disable can-do
  rule 'can-do' disabled (kept on record)
provenas> :why alice can prod_deploy
  no
provenas> :enable can-do
  rule 'can-do' re-enabled

This doubles as what-if analysis: "how much access exists only because of role inheritance?" Switch that rule off, ask again, compare, switch it back on. Answers update instantly.

4. Everything is on the record

:log shows the journal: every fact added or removed, every rule admitted or rejected, and every question answered — with the answer it gave. "What did the system say last Tuesday, and why?" has an actual answer.

05

Talk to it in English

Everything so far worked with typed commands. Connect an AI model and you can just… ask. The model's only job is translating your sentence into one of the exact commands you've already seen — the fact-keeper still does all the answering.

in your terminal — three ways to connect a model# Option 1: a free local model via Ollama (nothing leaves your machine)
ollama pull qwen3.5:9b

# Option 2: an Ollama server on another machine
export PROVENAS_LLM_HOST=http://my-server:11434

# Option 3: any OpenAI-style service (OpenAI, OpenRouter, LM Studio, …)
export PROVENAS_LLM_BACKEND=openai
export PROVENAS_LLM_HOST=https://api.openai.com
export PROVENAS_LLM_API_KEY=sk-...
export PROVENAS_LLM_MODEL=gpt-4o-mini
provenas — try thisprovenas> Can Alice deploy to production?
  → yes
    [{'action': 'check', 'triple': ['alice', 'can', 'prod_deploy']}]
    (alice can prod_deploy)   ⇐ rule[can-do]
      …the same receipt as chapter 02…

The dim line in brackets shows you the translation the model proposed — full transparency. If the model misunderstands your question, the worst case is an honest "not derivable," never an invented yes.

Teach it a rule by example

Describe the rule in English and give a few examples of right (and wrong) answers. The model drafts the rule; Provenas tests the draft: it must produce all your "should be true" examples and none of your "should be false" ones — and not flip any pinned answer from chapter 04. Pass → saved. Fail → the model gets the failure report, tries again, and if it still fails, nothing is saved.

provenas — try thisprovenas> :load family
provenas> :learn sibling : share a parent, different people :: ann,cy cy,ann | ann,ann ann,dan
  proposed: sibling ⇐ (?p parent ?x), (?p parent ?y), (?x != ?y)
  -> ADMITTED (saved)  (missing=[] violated=[])

(Before the |: pairs that should count as siblings. After it: pairs that should not — Ann isn't her own sibling.)

Let it write a small tool — inside a cage

provenas — try thisprovenas> :tool gcd : greatest common divisor of a and b :: gcd(12,8)=4, gcd(7,5)=1
  proposed:
      def gcd(a, b):
          if b == 0:
              return a
          return gcd(b, a % b)
  -> ADMITTED (saved): all 2 examples passed
provenas> :call gcd 48 36
  gcd(48, 36) = 12

AI-written code is the scariest kind, so it faces the strictest gate: the code is inspected for anything dangerous (no file access, no network, no sneaky tricks — rejected before it ever runs), then executed in a locked, time-and-memory-limited cage against your examples. Code that loops forever gets killed; code that's wrong fails the examples; only code that's both safe and correct is kept.

Honest note Smaller AI models occasionally garble a translation (for example, flipping "Cy's ancestors" into "Cy's descendants"). When that happens the answer comes back empty or "not derivable" — wrong-shaped, but never false — and a larger model fixes it. The testing gates are what make a weak model safe; they can't make it smart.
06

Let your apps ask it

The same knowledge base can run as a tiny web service on your machine, so your other programs (in any language) can ask it questions over HTTP — like a smart light switch for decisions. No AI model required for this mode at all.

in your terminalexport PROVENAS_API_TOKEN=sekrit        # optional password
provenas-serve mykb.db 8642
any program, anywhere — here using curlcurl -s -H "Authorization: Bearer sekrit" localhost:8642/action \
     -d '{"action":"check","triple":["alice","can","prod_deploy"]}'

{"kind": "check", "answer": true,
 "trace": "(alice can prod_deploy)   ⇐ rule[can-do]\n  (alice has_perm deploy_perm) …"}

The receipt (trace) rides along in every response — your app can show it to a user, attach it to a ticket, or store it for an audit.

addresswhat it does
GET /health"are you alive?" + fact/rule counts
POST /actionask or add, exactly — no AI involved
POST /ask{"q": "plain English"} → translated, then answered exactly (needs a model)
GET /facts?contains=xbrowse the stored facts
GET /rules · GET /schemathe active rules · the declared vocabulary

One process owns the file at a time — it's a personal sidecar, not a big shared database cluster, and it doesn't pretend to be one.

07

For Python programmers

If you write Python, the CLI and the web service are thin wrappers around a small library you can use directly. The whole surface fits in one screen:

pythonfrom provenas.store import Store
from provenas.infer import Rule
from provenas.learn import admit_rule
from provenas import qa

store = Store("mykb.db")                      # or ":memory:" for a throwaway
store.assert_("tom", "parent", "bob", source="import")
store.add_rule(Rule([("?x", "parent", "?y")], ("?x", "ancestor", "?y"), "anc"))

# ask — same dict shape the CLI and HTTP service use
r = qa.run_action(store, {"action": "check", "triple": ["tom", "ancestor", "bob"]})
r["answer"]   # True
r["trace"]    # the receipt, as text

# gate a candidate rule instead of trusting it
ok, report = admit_rule(store, candidate, "sibling",
                        positives=[("ann","cy")], negatives=[("ann","ann")])
callwhat it gives you
Store(path)the knowledge base file: assert_/retract/triples, add_rule/rules/set_rule_active, add_case/cases, declare/schema, closure(), log/recent_log
qa.run_actionquestion in → {kind, answer, trace} out; every decision logged
admit_ruletests a rule against your examples + pinned cases; saves only on pass
forward_chain(kg, rules)the bare engine, if you bring your own in-memory graph
admit_toolcode + examples → safety inspection → caged test run → saved
08

Real-life examples

Three complete, heavily-commented programs ship in the examples/ folder. Each runs offline in a few seconds — no AI model, nothing extra to install — and the test suite runs them too, so they always match the real behavior of the code.

python examples/access_audit.py — who can touch production, and why?

A realistic small company: roles that inherit from each other, permissions, resources, and one suspended contractor. It answers the questions an auditor actually asks — who can deploy? show me why. the contractor has the permission, so why is he blocked? — then runs a what-if (switch off the inheritance rule, see exactly whose access disappears), catches a typo'd fact with strict mode, and prints the decision log. Swap in an export from your real systems and this is a working access auditor.

python examples/eligibility_screening.py — who qualifies, and why?

Four applicants, three benefit rules straight out of a policy document: senior discount at 65+; income assistance under $30,000 unless disqualified; food support for households of 3+ under $45,000. Every verdict prints the receipt an appeal would demand. The finale: a denial is pinned as a case, then someone proposes a "simpler" rule that forgot the disqualified-clause — and the gate rejects it, because it would quietly re-approve the pinned denial.

python examples/service_sidecar.py — your app asks over HTTP

Starts the web service and then plays the role of your application: health check, adding facts, asking questions, getting receipts back in the response — including a clean, honest false for the engineer who lacks a deploy key. The little client() function in it is copy-paste integration for any Python codebase.

Suggested path Run all three. Then pick something you personally keep answering by hand — repo permissions, on-call rules, club membership tiers, equipment compatibility — export it as facts, write the three-to-ten rules, and you'll have the feel of the tool in an afternoon.

Cheatsheet & fine print

Command cheatsheet

commanddoes
:load <pack>load a ready-made set (family / rbac / diagnostics / eligibility / config)
:assert s r o · :retract s r oadd / remove a fact
:why s r o"is this true?" — with the receipt
:learn rel : description :: good pairs | bad pairsAI drafts a rule → tested → saved only if it passes
:tool name : description :: f(a)=r, …AI writes a function → inspected → caged test → saved
:call name args [as s r]run a saved tool; optionally store its result as a fact
:simplify (x + 0) * (2 + 3)exact algebra → x * 5
:case name · :casespin the last answer · list pinned answers
:disable name · :enable nameswitch a rule off/on (kept on record)
:declare rel [doc] · :schema · :strict on|offdeclare vocabulary; strict mode rejects undeclared words
:facts [text] · :rules · :tools · :rewrites · :log [n] · :kblook at everything
:narrate on|offoptional one-line spoken-style summary of each exact answer
:help · :quityou guessed it

The question format (used by the CLI, the web service, and the library)

{"action": "query",  "pattern": ["?x", "can", "prod_deploy"]}   # "who can deploy?" — ?x is the blank
{"action": "check",  "triple":  ["alice", "can", "prod_deploy"]} # "is this true?"
{"action": "assert", "triple":  ["dana", "has_role", "analyst"]} # "remember this."

The fine print (honest limits)

  • It only knows what you've taught it. Ask outside the knowledge base and you get "not derivable" — on purpose. It's a fact-keeper, not a search engine.
  • English understanding depends on the AI model you connect. The safety gates make a weak model safe; they can't make it smart.
  • The code cage is strong, not magical. It blocks dangerous code several ways, but if strangers will run code on your machine, use real OS-level isolation (containers) on top.
  • One machine, one writer. Comfortable up to hundreds of thousands of facts (there's a test enforcing that). It is not a distributed database and doesn't pretend to be.

That's the whole idea: facts in, rules over them, receipts out — and an AI that may only ever suggest. Go point it at something you're tired of answering by hand.
