The idea in 60 seconds

LLMs are wonderful at language and terrible at being trusted. Provenas splits the job in two.

An LLM proposes — it translates your question into a tiny structured action, drafts rules, drafts code. An exact symbolic fabric verifies and answers — a knowledge base of facts and logic rules, evaluated by a real Datalog engine, with SQLite underneath remembering everything. The model is the mouth and hands; the fabric is the source of truth.

Two consequences fall out of that split:

  • Every answer carries its derivation. Not "yes", but "yes, because these facts and these rules" — a proof tree down to the given facts.
  • Nothing is admitted without a test. A learned rule must derive every positive example and no negative one, must not flip any decision you've pinned, and a synthesized tool must survive static analysis and a sandboxed run. A wrong proposal is rejected; it never touches the store.

When the model mis-reads you, the fabric says "not derivable" — an honest no, never a confident invention. That property is the whole point.

01

Install & your first facts

The engine and CLI are pure Python standard library — no heavy dependencies.

shell# Python 3.10+
git clone https://github.com/calvincs/Provenas && cd Provenas
pip install -e .
provenas mykb.db        # opens (or creates) a persistent SQLite knowledge base

Everything in the knowledge base is a fact: a triple of (subject, relation, object). Add some, query them, and ask why:

provenas — the CLIprovenas> :assert tom parent bob
  added (tom parent bob)
provenas> :assert bob parent ann
  added (bob parent ann)
provenas> :facts
  (bob parent ann)
  (tom parent bob)
  [2 fact(s)]
provenas> :why tom parent bob
  yes
    (tom parent bob)   (given fact)
No model needed yet Every :command works completely offline. The English layer (chapter 05) is an optional add-on, never a requirement — and never the authority.

Don't want to type facts by hand? Five starter packs ship in the box: :load family, rbac, diagnostics, eligibility, or config.

02

Rules, derived facts, and proofs

Facts you state; rules derive what they imply. A rule is a Horn clause — "if all the body atoms hold, the head holds" — written with ?variables. Load the access-control pack and look at one:

provenasprovenas> :load rbac
  loaded 'rbac'
provenas> :rules
  role-inherit: ?u has_role ?r2  ⇐  (?u has_role ?r), (?r inherits ?r2)
  perm-from-role: ?u has_perm ?p  ⇐  (?u has_role ?r), (?r grants ?p)
  can-do: ?u can ?a  ⇐  (?u has_perm ?p), (?a requires ?p)
  [3 active / 3 rule(s)]  (:disable/:enable <name>)

The engine forward-chains those rules to a fixpoint — and materializes the result, so queries are answered from a cache that only recomputes when facts or rules change. Ask something the rules imply, and the proof tree comes back with it:

provenasprovenas> :why alice can prod_deploy
  yes
    (alice can prod_deploy)   ⇐ rule[can-do]
      (alice has_perm deploy_perm)   ⇐ rule[perm-from-role]
        (alice has_role engineer)   (given fact)
        (engineer grants deploy_perm)   (given fact)
      (prod_deploy requires deploy_perm)   (given fact)

Read it bottom-up: two given facts make alice has_perm deploy_perm; that plus a third fact makes alice can prod_deploy. This is what "explainable" means here — not a narrative, a derivation.

Scale, measured The evaluator is indexed, semi-naive, stratified Datalog: 50,000 base facts close to a 287,000-fact knowledge base in ~1.3 s; 200,000 → 1.15 M in ~6 s; reads from the materialized closure are sub-millisecond. tests/test_perf.py enforces the envelope.
03

Real policy: negation & comparisons

Real rules are rarely just lookups. They say "unless" and they compare numbers. Rule bodies support three kinds of atom:

atommeaning
["?u", "has_role", "?r"] Lookup — joins against the stored + derived facts.
["?u", "~suspended", "?x"] Negation (~rel) — holds iff no matching fact exists (NOT EXISTS). Evaluation is stratified: anything that can be suspended is fully derived before anything that depends on not being suspended. A negation cycle is rejected outright.
["?age", ">=", "65"] Guard — != == < <= > >=, compared numerically when both sides are numbers (ISO dates also order correctly as strings).

Which makes "allowed unless suspended" and "qualifies at 65+" one-liners:

python — rules as dataRule([("?u", "can", "?a"), ("?u", "~suspended", "?when")],
     ("?u", "allowed", "?a"), "allowed")          # capable AND not suspended

Rule([("?p", "age", "?a"), ("?a", ">=", "65")],
     ("?p", "eligible_for", "senior_discount"), "senior-65plus")

And the proof still tells the whole story — the contractor below is capable of pushing to main, but not allowed, and the system can show you both:

examples/access_audit.py (real output)[audit] Is the contractor CAPABLE of pushing to main?  (raw permission)
        -> yes
[audit] ...but is he ALLOWED?  (suspension wins — negation, not a special case)
        -> no
        (dev_dan allowed main_branch) is not derivable from the facts + rules.
04

Guardrails: cases, schema, history, log

This is the chapter that makes it production-shaped rather than a demo.

Pin decisions — CI for rules

After any answer, pin it. From then on, every rule change is replayed against your pinned cases, and a rule that flips one is rejected — even if its own examples pass:

provenasprovenas> :why alice can prod_deploy
  yes …proof…
provenas> :case alice-prod
  pinned case 'alice-prod': {'action': 'check', 'triple': ['alice', 'can', 'prod_deploy']} -> True
  (rule changes must preserve it)

Strict schema — a typo can't corrupt the data

provenasprovenas> :strict on
  strict mode ON — asserts with undeclared relations are rejected
provenas> :assert bob hsa_role intern
  error: relation 'hsa_role' is not declared in the schema (strict mode)

Starter packs declare their relations automatically; declare your own with :declare has_role "person holds role".

Rules have history, not just state

provenasprovenas> :disable can-do
  rule 'can-do' disabled (kept on record)
provenas> :why alice can prod_deploy
  no
provenas> :enable can-do
  rule 'can-do' re-enabled

The materialized closure updates the moment a rule's status changes — that's also your what-if analysis: disable a rule, see exactly which access or eligibility evaporates, re-enable it.

The decision log

The audit log (:log) records every change and every decision — each query/check with its answer — so "what did the system say last Tuesday and why" has an answer.

05

The English layer

Connect any chat model and the same knowledge base becomes conversational. The model only ever emits a small JSON action (query / check / assert); the fabric executes it exactly.

shell — three ways to connect a model# 1. Local Ollama (the default — nothing to set)
ollama pull qwen3.5:9b

# 2. Remote Ollama server
export PROVENAS_LLM_HOST=http://my-server:11434
export PROVENAS_LLM_MODEL=qwen3.5:9b

# 3. Any OpenAI-compatible API (OpenAI, OpenRouter, vLLM, LM Studio, llama.cpp, …)
export PROVENAS_LLM_BACKEND=openai
export PROVENAS_LLM_HOST=https://api.openai.com
export PROVENAS_LLM_API_KEY=sk-...
export PROVENAS_LLM_MODEL=gpt-4o-mini
provenasprovenas> Can Alice deploy to production?
  → yes
    [{'action': 'check', 'triple': ['alice', 'can', 'prod_deploy']}]
    (alice can prod_deploy)   ⇐ rule[can-do]
      …the full proof, as in chapter 02…

Teach it a rule — through the gate

:learn asks the model to draft a rule from your description and ± example pairs. The gate tests it; if it fails, the verdict is fed back and the model revises; if it still fails, it's rejected. Only a verified rule is ever committed.

provenasprovenas> :load family
provenas> :learn sibling : share a parent, different people :: ann,cy cy,ann | ann,ann ann,dan
  proposed: sibling ⇐ (?p parent ?x), (?p parent ?y), (?x != ?y)
  -> ADMITTED (saved)  (missing=[] violated=[])

Grow a tool — safe and correct, or not at all

provenasprovenas> :tool gcd : greatest common divisor of a and b :: gcd(12,8)=4, gcd(7,5)=1
  proposed:
      def gcd(a, b):
          if b == 0:
              return a
          return gcd(b, a % b)
  -> ADMITTED (saved): all 2 examples passed
provenas> :call gcd 48 36
  gcd(48, 36) = 12

LLM-written code is admitted only after an AST allowlist (no imports, no attribute access, no dangerous builtins) and a resource-limited sandboxed run against your examples. An import os dies statically; an infinite loop is killed by the sandbox; a wrong implementation fails the tests.

Honest note Translation quality tracks the model. A small model occasionally inverts a relation's direction — the fabric then answers "not derivable" rather than something false, and a stronger model (PROVENAS_LLM_MODEL=qwen3.5:27b-q4_K_M) fixes it. The gates are what make a weak model safe, not smart.
06

The decision sidecar

Run the same knowledge base as a local HTTP service — the way OPA runs beside an app, except every answer carries its proof. POST /action needs no model at all.

shellexport PROVENAS_API_TOKEN=sekrit        # optional bearer auth
provenas-serve mykb.db 8642
your service (any language)curl -s -H "Authorization: Bearer sekrit" localhost:8642/action \
     -d '{"action":"check","triple":["alice","can","prod_deploy"]}'

{"kind": "check", "answer": true,
 "trace": "(alice can prod_deploy)   ⇐ rule[can-do]\n  (alice has_perm deploy_perm) …"}
endpointdoes
GET /healthliveness + fact/rule counts + model reachability
POST /actionexact query/check/assert — no model needed
POST /ask{"q": "plain English"} → translate → exact answer (needs a model)
GET /facts?contains=xbrowse stored facts
GET /rules · GET /schemaactive rules · declared relations + strict flag

Single-writer by design: one process owns the SQLite file; the server serializes access (WAL + lock). It's a sidecar, not a distributed database.

07

The Python API

Everything the CLI does is a thin layer over a small library. The five objects that matter:

pythonfrom provenas.store import Store
from provenas.infer import Rule
from provenas.learn import admit_rule
from provenas import qa

store = Store("mykb.db")                      # or ":memory:"
store.assert_("tom", "parent", "bob", source="import")
store.add_rule(Rule([("?x", "parent", "?y")], ("?x", "ancestor", "?y"), "anc"))

# the canonical answer path — same dict the CLI and HTTP service return
r = qa.run_action(store, {"action": "check", "triple": ["tom", "ancestor", "bob"]})
r["answer"]   # True
r["trace"]    # the proof tree, as text

# gate a candidate rule instead of trusting it
ok, report = admit_rule(store, candidate, "sibling",
                        positives=[("ann","cy")], negatives=[("ann","ann")])

# the materialized closure directly (read-only)
kg, prov = store.closure()
callwhat it gives you
Store(path)the persistent KB: assert_/retract/triples, add_rule/rules/set_rule_active, add_case/cases, declare/schema, closure(), log/recent_log
qa.run_actionaction in → {kind, answer, trace} out; decisions logged
admit_rule± examples + pinned-case replay; commits only on pass
forward_chain(kg, rules)the raw engine, if you bring your own KnowledgeGraph
admit_tool (toolsmith)source + examples → AST gate → sandbox → registry
08

Worked real-life examples

Three complete, narrated programs ship in examples/. Each runs offline in seconds — no model, no third-party packages — and CI runs them, so they can't rot.

python examples/access_audit.py — the access auditor

A realistic engineering org (roles, inheritance, grants, a suspended contractor) answering the audit questions: who can deploy and why; capable vs allowed; what access exists only through inheritance (a what-if via :disable); strict schema catching a typo'd relation; the decision log. This is the "point it at your IAM export" blueprint.

python examples/eligibility_screening.py — benefits screening

Four applicants, three program rules using >=/< guards and ~disqualified. Every verdict carries the derivation an appeal would demand — and the finale pins a denial as a regression case, then watches the gate reject a sloppy rewrite that would have silently re-approved it.

python examples/service_sidecar.py — the HTTP sidecar

Starts the service and queries it the way your app would: health, asserts, checks with proofs in the response, an honest false for the engineer without a deploy key. Copy the client() function into any codebase and you've integrated.

Suggested path Run all three, then swap the seeded facts for a CSV/JSON export of something you actually manage — repo permissions, on-call rules, device inventory. The rules rarely number more than a dozen.

Honest limits

  • It only knows what you've asserted. Out-of-KB questions get "not derivable" — by design. It complements retrieval/RAG; it doesn't replace it.
  • NL accuracy tracks the model. The gates make weak models safe, not smart.
  • The sandbox is defense-in-depth (AST allowlist + rlimited subprocess), not OS-level isolation. Untrusted multi-tenant code needs containers/seccomp.
  • Single-writer, one box. The service serializes access; envelope is 10⁵–10⁶ facts, enforced by the perf tests. It is not Soufflé and not a cluster.

That's the whole tool: facts in, rules over them, proofs out — and a model that may only ever propose. Go point it at something you're tired of answering by hand.
